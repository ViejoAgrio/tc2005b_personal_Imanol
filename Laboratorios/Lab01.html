¿Cuáles son las partes de un URL?

Una URL (Uniform Resource Locator) es una dirección identificadora única que permite encontrar recursos en Internet, como páginas web, archivos o documentos. Las partes principales de una URL son Protocolo (HTTPS), el cual define cómo se transferirá la información entre diferentes actores en la World Wide Web, Subdominio (www), es lo que aparece entre el protocolo y el primer punto que separa el dominio, Dominio (ejemplo.com), compuesto por dos elementos: el nombre de dominio y la TLD, Ruta (/subejemplo/ruta), que indica la ubicación específica del recurso dentro del dominio.

Source: Conversation with Bing, 5/4/2024 (1) Partes de una URL con ejemplos y explicación fácil - Edyta Pukocz. https://edytapukocz.com/url-partes-ejemplos-facil/.

¿Cuál es el propósito de los métodos HTTP: GET, HEAD, POST, PUT, PATCH, DELETE?

GET: Solicita una representación de un recurso específico.

HEAD: Pide una respuesta idéntica a la de una petición GET, pero sin el cuerpo de la respuesta.

POST: Envia una entidad (datos) a un recurso específico en el servidor.

PUT: Reemplaza todas las representaciones actuales del recurso de destino con la carga útil de la petición.

PATCH: Aplica modificaciones parciales a un recurso.

DELETE: Borra un recurso específico.

Source: Conversation with Bing, 5/4/2024 (1) Métodos de petición HTTP - HTTP | MDN - MDN Web Docs. https://developer.mozilla.org/es/docs/Web/HTTP/Methods.

¿Qué método HTTP se debe utilizar al enviar un formulario HTML, por ejemplo cuando ingresas tu usuario y contraseña en algún sitio? ¿Por qué?

POST: Al enviar datos sensibles como contraseñas, es importante que no se muestren en la URL (como ocurre con GET). POST envía los datos en el cuerpo de la solicitud, lo que es más seguro.

Source: Conversation with Bing, 5/4/2024 (1) Formulario HTML – Ejemplos de tipo de entrada y botón de envío. https://www.freecodecamp.org/espanol/news/formulario-html-ejemplos-de-tipo-de-entrada-y-boton-de-envio/.

GET: Cuando ingresas una URL en tu navegador y presionas “Enter”, estás realizando una solicitud GET al servidor web para obtener la página correspondiente.

Un servidor web devuelve una respuesta HTTP con código 200. ¿Qué significa esto? ¿Ocurrió algún error?

Cuando visitas una página web, estás realizando una solicitud GET. Si recibes una respuesta con código 200, significa que la página se ha cargado correctamente.

¿Es responsabilidad del desarrollador corregir un sitio web si un usuario reporta que intentó acceder al sitio y se encontró con un error 404? ¿Por qué?

El error 404 indica que tu navegadorse ha conectado con un servidor, pero este no puede mostrarte la web que deseas. Las principales causas con cambio de nombre o dirección, desparición de la URL, errores al escribir la dirección URL o cambios de servidor. Por lo tanto, salvo por la causa 3, son problemas que los desarrolladores tienen la responsabilidad de resolver.

Source: Conversation with Bing, 5/4/2024 (1) Error 404: ¿qué significa y cómo solucionarlo? | Computer Hoy. https://computerhoy.com/tutoriales/error-404-significa-como-solucionarlo-1284880.

¿Es responsabilidad del desarrollador corregir un sitio web si un usuario reporta que intentó acceder al sitio y se encontró con un error 500? ¿Por qué?

El Error 500 es un código de estado HTTP que indica que algo ha salido mal en el servidor web al intentar procesar una solicitud del navegador. Es una respuesta genérica que el servidor proporciona cuando se encuentra con una situación inesperada que no le permite cumplir con la solicitud. Las principales causas son errores en el código de la aplicación web o en los scripts que el servidor ejecuta o configuraciones incorrectas o incompatibles en el servidor web, por lo que en efecto es responsabilidad del desarrollador solucionarlo.

Source: Conversation with Bing, 5/4/2024 (1) Error 500: ¿qué significa y cómo solucionarlo? | Computer Hoy. https://computerhoy.com/tutoriales/error-500-significa-como-solucionarlo-1293380.

¿Cuál es el ciclo de vida de los sistemas de información?

El ciclo de vida de un sistema de información es un modelo estructurado de gestión de proyectos que describe las fases necesarias para construir un sistema de información, desde el inicio hasta el resultado final. Las siete etapas son: Planificación, análsiis de requerimientos, diseño, desarrollo, evaluación, entrega y mantenimiento.
